## 1. What this project is (one paragraph)

RealLearn AI is an AI-native learning platform. A learner asks one question,
picks a language (12 Indian languages), a level (Class 6-8 / 9-10 / College),
and a mode (Fast = one direct answer, Explain = a 3-part journey:
Foundation → Mechanism → Real World). Each part is gated behind a 2-question
quiz you must score 100% on to advance. Lessons are generated by Gemma 4
(Cerebras primary, Cloudflare fallback), grounded with live Serper news in
Part 3, streamed over SSE, and cached in a two-tier LRU + MongoDB cache. The
aesthetic is the **Soft Pastel** design language — cozy and colorful without
costume: cream paper washed with pastel sky, peach, mint and rose; a clear
sky-blue accent anchors buttons and links (NO purple/violet — owner's rule).
Three moods of the same room: Paper (cream day), Ink (charcoal night), Dusk
(teal sunset evening). Frontend: Next.js 15 + React 19 + TypeScript +
Tailwind + Clerk + Zustand. Backend: Node + Express.

## 5. Design system — USE THE CLASSES, NOT INLINE STYLES

`frontend/app/globals.css` defines a real design system. **The #1 thing that
makes this codebase look AI-generated is `style={{ ... }}` soup in components**
instead of using these classes.** Every interactive surface should use them.**

Available component classes (already implemented — use them):
- Buttons: `.btn-primary`, `.btn-ghost`, `.btn-icon`, `.btn-icon--danger`
- Nav: `.nav-link`, `.nav-link--active`
- Chips: `.chip`, `.chip__label`, `.chip__value`
- Headers: `.section-header`, `.section-overline`, `.section-header__kanji`,
  `.section-header__title`, `.section-header__subtitle`
- Cards: `.rl-card`, `.stat-tile`
- Learning: `.part-card`, `.part-cta`, `.part-done-bar`, `.progress-hub`
- Quiz: `.quiz-sheet`, `.quiz-sheet__panel`, `.quiz-sheet__close`, `.quiz-sheet__action`,
  `.quiz-question`, `.quiz-question__option`, `.quiz-question__badge`, `.quiz-question__explanation`
- Loading / error: `.loading-cinematic`, `.loading-cinematic__progress`, `.loading-cinematic__steps`,
  `.loading-cinematic__cancel`, `.error-state`, `.error-state__card`, `.error-state__btn`
- Toggles: `.btn-toggle`, `.mode-glider`
- Decorative: `.kusari` (chain divider), `.liquid-sheen`, `.engraved`,
  `.identity-texture`, `.texture-noise`, `.texture-dots`
- Misc: `.scroll-top`, `.hero-greeting`, `.hero-greeting-icon`, `.app-footer`

**Rules:**
- New layout = add a class to `globals.css`, then use it. Do not inline a
  15-property style object.
- Spacing/radius/shadow/color → use the CSS tokens (`var(--space-*`,
  `var(--radius-*)`, `var(--shadow-*)`, `var(--text-primary)`, `var(--accent)`,
  `var(--bg-card)`, …). Never hardcode hex in components.
- Inline styles are acceptable **only** for genuinely dynamic/computed values
  (e.g. a `translateX` driven by state, a `width` from a measurement). Not for
  static padding/colors/fonts.

## 6. "De-slop" rules — how to not look AI-generated

These are the tells we are actively removing. Apply them on every UI touch:

1. **No flowery JSDoc/inline comments that over-explain trivia.**
   Bad: `/** Warm, time-aware greeting — the app says hello like a friend would. */`
   Good: `// time-of-day greeting`
   Comments explain *why*, not *what*. Delete comments that paraphrase the code.
2. **Restrain emoji.** A single contextual emoji can be fine; emoji-every-greeting
   (🌙☀️🌆✨) reads as AI. Default to no decorative emoji in UI chrome.
3. **Human microcopy.** Short, plain, specific. Avoid marketing superlatives
   ("unforgettable learning journey", "teaching, reimagined for the AI era").
   The product copy in `llms-full.txt` is for *brand docs*, not for the UI.
4. **One accent, used sparingly.** Sky blue is the accent (pastels are for washes). Don't gradient-text
   everything. Reserve gradient text for the wordmark and one hero element.
5. **Prefer composition over giant components.** A 280-line `page.tsx` with
   nested inline-style divs is the smell. Extract sub-pieces.
6. **No "celebration" noise unless it serves the moment.** Confetti on quiz
   pass is intentional (the user chose it); confetti/emoji on every micro-
   interaction is slop.
7. **Consistency > novelty.** Don't invent a new button style for one screen.
8. **Comments in English, concise, no AI-signature phrases** ("delve into",
   "in this comprehensive", "let's", "it's worth noting", "seamless",
   "robust", "cutting-edge", "leveraging"). If you'd write it in a ChatGPT
   answer, rewrite it.

## 7. UX principles for this product

1. **The primary action must always be obvious.** On the homepage that is the
   question input + submit. Don't bury it under decoration.
2. **Every async state has a real UI:** loading, empty, error, success.
   Never leave a user staring at nothing.
3. **Keyboard-first, touch-friendly.** 44px min touch targets, visible focus,
   Enter/Cmd+Enter to submit, arrow keys in quizzes, focus trap in modals.
4. **Respect `prefers-reduced-motion`.** All animations gate on it.
5. **Mobile is not an afterthought.** Test at 360px. Nav collapses to sidebar
   below 900px (by design).
6. **Honest about AI.** Keep the "AI-generated, may be inaccurate" disclaimers.
7. **Don't add features to "improve" UX without checking it isn't already
   there.** This repo is mature (586+ commits). Search before you build.

## 8. Accessibility bar

Target: WCAG 2.1 AA. Non-negotiables:
- Color contrast ≥ 4.5:1 for text (tokens already meet this — don't regress).
- All icon-only buttons have `aria-label`.
- Modals trap focus and restore it on close; Escape closes when safe.
- Live regions (`aria-live`) for toasts, loading, quiz feedback.
- `prefers-reduced-motion` respected everywhere.
- Skip-to-content link present.

## 9. Known no-gos (don't do these)

- Don't hardcode the backend URL. Use `process.env.NEXT_PUBLIC_BACKEND_URL`
  (fallback `https://real-learn.onrender.com`).
- Don't bypass the quiz gate — 100% pass to unlock the next part is the product.
- Don't machine-translate lessons. Generation is natively multilingual.
- Don't put secrets in the frontend. `.env.local.example` is the template.
- Don't remove the AI-content disclaimers or the consent/legal flows.
- Don't add a new AI provider without the circuit-breaker/retry pattern.
- Don't commit `node_modules`, `.next/`, or `.env*` (they're gitignored — keep it that way).

## 11. CHANGE PROTOCOL — "if you do even a small change or a big change"

This is the rule the repo owner set. **Every change, small or big, triggers
this protocol. No exceptions.**

1. **Read this file first.** (You're doing that now.)
2. **Branch** off `main` (see §4).
3. **Make the change.** Use the design system (§5), follow de-slop rules (§6)
   and UX principles (§7) for any UI work.
4. **Verify:** `npx tsc --noEmit` + `npx next lint` + `npm run build` (frontend)
   or `npm test` (backend). All must pass.
5. **Update the docs that the change affects:**
   - Always append a one-line entry to `change-made-after-submission.md`
     (date + short description; match the existing table format).
   - If the change affects architecture/conventions/commands/no-gos → update
     **this file** (`docs/AGENT_MEMORY.md`).
   - If the change affects public brand facts → update `llms.txt` +
     `llms-full.txt`.
   - If the change affects the design system → update `DESIGN_AUDIT.md` and/or
     `IMPROVEMENT_PRIORITIES.md` (mark items done).
6. **Commit** with a conventional, specific message. Stage the doc updates in
   the same commit or a tightly-coupled follow-up commit.
7. **Push + open a PR.** In the PR body, explicitly list which docs you updated.
8. **Never mark a task done in any todo file without concrete evidence** (a
   passing build, a screenshot, a green test).

> Small change = still do steps 1, 2, 4, 5 (at least the changelog line), 6, 7.
> Big change = do all of the above, plus update this memory file.
